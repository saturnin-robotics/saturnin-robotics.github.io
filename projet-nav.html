Autonomous Navigation & SLAM on TurtleBot 2

2025 TurtleBot 2 ROS 2 Humble SLAM & Nav2
ROS 2 Humble Nav2 slam_toolbox Gazebo RViz 2 C++ robot_localization

This project implements a complete autonomous navigation pipeline on a TurtleBot 2 platform within ROS 2. The objective was to achieve reliable point-to-point navigation in a cluttered lab environment, managing both mapping (offline) and real-time obstacle avoidance. The work served as a concrete validation platform for sensor fusion and SLAM techniques studied during my M.Sc.

Technical Challenges

  • Sensor Fusion & Odometry Drift: Wheel odometry alone accumulates significant drift over long corridors. I integrated an IMU and fused it with wheel encoders using robot_localization (Extended Kalman Filter), reducing positional error by ~40% over a 20 m trajectory.
  • Mapping: Used slam_toolbox in asynchronous mode to build an occupancy grid map of the lab. Loop closure detection was critical in corridors with repetitive structure.
  • Navigation Parameter Tuning: Tuning the costmap inflation radius and the DWB local planner parameters was non-trivial — a radius too large prevented passage through 80 cm doorways, too small caused wall-grazing. Final solution: dynamic inflation layers.
  • Behavior Trees: Implemented custom BT nodes for recovery behaviors (rotate, back-up) when the robot became trapped in local minima.

System Architecture

The stack runs across two compute nodes: a Raspberry Pi 4 handles sensor drivers and base control; a remote workstation runs Nav2 and RViz 2 via the ROS 2 DDS network.

// Pipeline Overview
LiDAR + IMU + Odometry → robot_localization (EKF) → slam_toolbox → Nav2 (Global + Local Planner) → Kobuki Base 
# Launch the full navigation stack
ros2 launch my_bot_nav bringup_launch.py map:=/maps/lab_map.yaml

# Monitor the costmap in real-time
ros2 run rviz2 rviz2 -d config/nav_debug.rviz

Results

✓ Outcome

The robot achieved a 95% navigation success rate across 40 trials in the lab environment, correctly avoiding dynamic obstacles (people walking) in real-time with a replanning latency under 120 ms. Localization error remained below ±8 cm (RMSE) on a 30 m loop.

Key lesson: the bottleneck was not the planner but sensor calibration — an improperly calibrated LiDAR-to-base_link transform caused persistent map misalignment until corrected via careful extrinsic calibration.
